# פרויקט מסכם בקריפטוגרפיה: תקשורת היברידית מאובטחת ב-Python

פרויקט זה מדגים מימוש של ערוץ תקשורת מאובטח מקצה לקצה בין שרת TCP ללקוח GUI (מבוסס Tkinter). הפרויקט מיישם **הצפנה היברידית** המשלבת מנגנונים סימטריים וא-סימטריים, תוך שימוש ברשות אישורים פיקטיבית (Fake CA) למניעת התקפות אדם-באמצע (Man-in-the-Middle).

## 📂 מבנה הקבצים בפרויקט

1.  **[crypto_utils.py](file:///c:/Users/aviad/kripto/crypto_utils.py)** - תשתיות קריפטוגרפיות בסיסיות (הצפנת AES-CBC, יצירת מפתחות ופענוח RSA, גיבוב SHA-256 וחתימות RSA-PSS). הקוד חוסם שימוש באלגוריתמים חלשים כגון SHA-1.

6.  **[cats/](file:///c:/Users/aviad/kripto/cats/)** - תיקיית משאבים המכילה 3 תמונות חתולים לניסוי העברת קבצים בינאריים.

## ⚡ הרצה מהירה של הכל יחד (באמצעות פקודה אחת)

כדי לפשט את תהליך ההרצה, לבדוק את הפרויקט ולצפות בלוגים של השרת והלקוח יחד, נכתב סקריפט הרצה אוטומטי `run_all.py`. 

הריצו בטרמינל את הפקודה הבאה:
```bash
python run_all.py
```

**מה קורה כאשר מריצים את הפקודה הזו?**
1. **אתחול CA**: הסקריפט מריץ את `ca_manager.py` שיוצר את מפתחות ה-CA (`ca_private.pem`, `ca_public.pem`).
2. **פתיחת מסוף השרת**: הסקריפט פותח חלון CMD/מסוף חדש שבו רץ השרת (`server.py`). חלון זה משמש כ-**Console** של השרת, ומציג בעברית בצורה מפורטת מאוד כל שלב בתקשורת (למשל: פענוח מפתח ה-AES, תהליך האימות, הצפנת התמונות וכו').
3. **הפעלת ממשק הלקוח**: לאחר המתנה של 1.5 שניות לעליית השרת, הסקריפט מפעיל אוטומטית את הלקוח הגרפי (`client_gui.py`).

## 🚀 מדריך הרצה מקיף (סדר פעולות מדויק)

כדי להריץ את הפרויקט בצורה תקינה, יש לעקוב בדיוק אחר השלבים הבאים:

### שלב 1: התקנת ספריות נדרשות
פתחו את הטרמינל והריצו את הפקודה הבאה להתקנת הספריות הדרושות (`cryptography` לקריפטוגרפיה ו-`Pillow` לעיבוד תמונות ב-Tkinter):
```bash
pip install cryptography pillow
```

### שלב 2: אתחול ויצירת מפתחות ה-CA
לפני שמפעילים את השרת או הלקוח, יש לייצר את המפתחות של רשות האישורים (CA). הריצו את הפקודה:
```bash
python ca_manager.py
```
**מה קורה ברקע?**
*   נוצרים הקבצים `ca_private.pem` (המפתח הפרטי של ה-CA) ו-`ca_public.pem` (המפתח הציבורי של ה-CA).
*   מבוצעת בדיקת תקינות המדמה יצירת תעודה ואימות שלה, וכן ניסיון זיוף תעודה שנחסם בהצלחה.

### שלב 3: הפעלת השרת
לאחר שיש לנו תשתיות CA, נפעיל את השרת שיאזין לחיבורים. הריצו:
```bash
python server.py
```
**מה קורה ברקע?**
*   השרת בודק אם יש לו מפתחות RSA משלו (`server_private.pem` ו-`server_public.pem`). אם לא, הוא מייצר אותם.
*   השרת פונה אוטומטית ל-`ca_manager` ומבקש לחתום על המפתח הציבורי שלו. נוצרת התעודה הדיגיטלית `server_cert.json`.
*   השרת עולה ומאזין בכתובת `127.0.0.1:8085`.

### שלב 4: הפעלת הלקוח (GUI)
בזמן שהשרת רץ ברקע, פתחו חלון טרמינל חדש והריצו את הלקוח:
```bash
python client_gui.py
```
**כיצד לבדוק את המערכת בממשק הגרפי?**
1.  **בחירת הצפנה**: בחרו בתיבת הבחירה (Combobox) את רמת ההצפנה הרצויה: `AES-256` או `AES-128`.
2.  **לחיצת יד**: לחצו על כפתור **"התחבר ובצע Handshake"**.
    *   ביומן האירועים (Secure Log) תראו את ההתקדמות בלייב: התחברות לשרת, קבלת התעודה, אימות חתימת התעודה מול `ca_public.pem`, הגרלת מפתח AES, הצפנתו ב-RSA ושליחתו לשרת.
    *   בסיום השלב, שדות ההזדהות וכפתור "התחבר לשרת" יהפכו לפעילים.
3.  **הזדהות**:
    *   נסו תחילה להזין שם משתמש או סיסמה שגויים ולחצו על **"התחבר לשרת"** כדי לראות שהשרת חוסם את הגישה ומציג שגיאה אדומה בלוג.
    *   הזינו את פרטי ברירת המחדל התקינים (שם משתמש: `aviad`, סיסמה: `123456`) ולחצו על **"התחבר לשרת"**.
4.  **פיענוח והצגת תמונות**:
    *   לאחר אימות תקין, השרת ישלח את 3 התמונות המוצפנות ב-AES.
    *   הלקוח יקרא את הבתים המוצפנים, יפענח אותם באמצעות מפתח ה-AES, ויציג את תמונות החתולים בתוך הממשק עם הכיתוב הירוק `[מפוענח ב-AES]`.

## 📝 תסריט הצגה לבוחן (Cheat Sheet)

להלן שלוש השאלות המרכזיות שהבוחן עשוי לשאול, עם התשובות הטכניות והקריפטוגרפיות המדויקות שעליכם להציג כדי להגן על הפרויקט:

### א. מהו תהליך לחיצת היד (Handshake) ההיברידי שבנינו, ולמה חייבים לשלב גם RSA וגם AES יחד?
**הסבר קצר ותמציתי:**
*   **הבעיה**: הצפנה א-סימטרית (RSA) היא איטית מאוד ומוגבלת בגודל המידע שניתן להצפין איתה (לא ניתן להצפין איתה קבצים גדולים כמו תמונות). מנגד, הצפנה סימטרית (AES) היא מהירה במיוחד ויעילה לקבצים גדולים, אך יש לה בעיית הפצת מפתח – איך שני הצדדים יסכימו על מפתח סודי משותף בלי שמאזין בתווך יגנוב אותו?
*   **הפתרון (הצפנה היברידית)**:
    1.  משתמשים ב-**RSA (א-סימטרי)** רק בשלב ה-Handshake כדי להעביר בבטחה את מפתח ה-AES. הלקוח מגריל מפתח AES אקראי, מצפין אותו עם המפתח הציבורי של השרת, ושולח אותו לשרת. רק השרת (שיש לו את המפתח הפרטי RSA התואם) יכול לפענח ולקבל את מפתח ה-AES.
    2.  משתמשים ב-**AES (סימטרי)** לכל אורך יתר השיחה – הצפנת פרטי ההתחברות והעברת התמונות הבינאריות. בכך אנו נהנים משני העולמות: אבטחה של הפצת מפתח (RSA) ומהירות הצפנה של קבצים (AES).

### ב. מהו מנגנון תעודות ה-CA? למה אי אפשר לסמוך רק על המפתח הציבורי של השרת, ואיך החתימה של ה-CA מונעת התקפת Man-in-the-Middle?
**הסבר קצר ותמציתי:**
*   **למה אי אפשר לסמוך ישירות על המפתח הציבורי שהשרת שולח?**
    אם השרת היה שולח את המפתח הציבורי שלו ללא תעודה חתומה, תוקף בתווך (MitM) היה יכול ליירט את המפתח הציבורי של השרת, להחליף אותו במפתח ציבורי משלו (של התוקף) ולשלוח אותו ללקוח. הלקוח היה מצפין את מפתח ה-AES שלו עם המפתח של התוקף, והתוקף היה יכול לפענח ולקרוא את כל התקשורת.
*   **איך החתימה של ה-CA מונעת זאת?**
    1.  רשות אישורים (CA) היא גורם שלישי מהימן (Trusted Third Party). ה-CA חותם על המפתח הציבורי של השרת באמצעות המפתח הפרטי של ה-CA.
    2.  החתימה מבוצעת על ידי גיבוב (Hash SHA-256) של שדות התעודה (שם השרת, מפתח ציבורי, שם המנפיק) והצפנת ה-Hash עם המפתח הפרטי של ה-CA (באמצעות RSA-PSS).
    3.  הלקוח מחזיק מראש ובצורה בטוחה את המפתח הציבורי של ה-CA (`ca_public.pem`).
    4.  כשהלקוח מקבל את התעודה, הוא מייצר מחדש את ה-Hash של השדות ומאמת את החתימה בעזרת מפתח ה-CA הציבורי שברשותו.
    5.  אם תוקף ינסה לשנות את המפתח הציבורי בתעודה או ליצור תעודה מזויפת, אימות החתימה ייכשל מכיוון שלתוקף אין גישה למפתח הפרטי של ה-CA.

### ג. איך פתרנו את נושא העברת קבצים בינאריים (תמונות) דרך שקעים (Sockets) בצורה מוצפנת?
**הסבר קצר ותמציתי:**
*   **בעיית ה-Fragmentation ב-TCP Sockets:**
    פרוטוקול TCP הוא פרוטוקול מבוסס זרם (Stream-oriented). כשקוראים מידע מה-Socket באמצעות פקודת `recv`, אין ערובה שנקבל את כל הבתים של התמונה בבת אחת (במיוחד קבצים בינאריים גדולים) או שלא נקבל מספר הודעות משורשרות יחד.
*   **הפתרון (הודעות עם אורך מוגדר מראש - Length-Prefixed Messages):**
    1.  לפני כל שליחת הודעה או קובץ תמונה מוצפן, השרת מחשב את גודל המידע המוצפן בבתים.
    2.  השרת שולח תחילה כותרת (Header) קבועה בגודל **4 בתים** המכילה את אורך המידע (Integer מיוצג בבתים בפורמט Network Byte Order באמצעות `struct.pack('!I', ...)`).
    3.  השרת שולח מיד לאחר מכן את בתים של התמונה המוצפנת.
    4.  הלקוח קורא מה-Socket בדיוק 4 בתים, מפענח את האורך המצופה, ואז מפעיל לולאה ייעודית שקוראת מה-Socket שוב ושוב עד שנאסף בדיוק מספר הבתים המבוקש.
    5.  רק לאחר איסוף כל המידע המוצפן של התמונה הספציפית, המידע מועבר לפענוח AES ועיבוד תמונה ב-Pillow. פתרון זה מונע אובדן מידע, ערבוב בין תמונות או קריסה של הפענוח.
